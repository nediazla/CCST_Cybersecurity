## 2. Explicación técnica clara y aplicada

### 1. Protocolos de transporte

#### TCP (Transmission Control Protocol)

- Confiable
    
- Orientado a conexión
    
- Usa el 3-way handshake (SYN, SYN-ACK, ACK)
    
- Garantiza entrega ordenada
    

Ataques comunes:

- SYN flood (muy preguntado)
    

#### UDP (User Datagram Protocol)

- No confiable
    
- Sin conexión
    
- Más rápido
    
- Usado en streaming, VoIP, DNS
    

Ataques comunes:

- UDP flood
    
- Amplificación (DNS, NTP)
    

#### ICMP

Usado para diagnóstico (ping, traceroute).  
No está diseñado para seguridad.

Ataques comunes:

- Ping flood
    
- Smurf attack
    

---

### 2. Protocolos de aplicación (relevancia directa al examen)

#### HTTP (puerto 80)

No cifrado.  
Vulnerable a:

- MITM
    
- Sniffing
    
- Manipulación
    

#### HTTPS (puerto 443)

HTTP + TLS  
Proporciona:

- Confidencialidad
    
- Integridad
    
- Autenticidad
    

Cisco pregunta mucho sobre HTTPS vs HTTP.

#### DNS (puerto 53)

Traduce dominios a IP.  
Vulnerabilidades:

- DNS spoofing
    
- DNS cache poisoning
    
- DNS tunneling
    

#### DHCP (puerto 67/68)

Asigna IP automáticamente.  
Riesgos:

- DHCP rogue
    
- Asignación de gateway/dns malicioso
    

#### FTP (puerto 21)

Inseguro; credenciales en texto claro.  
Alternativa segura: SFTP / FTPS.

#### SSH (puerto 22)

Acceso remoto seguro.  
Muy atacado por fuerza bruta.

#### SMTP (puerto 25)

Envío de correo.  
Riesgos:

- Spoofing
    
- Relay abierto
    

#### SNMP (puerto 161)

Administración de dispositivos de red.  
Riesgo:  
SNMPv1/v2 envían community strings en texto claro.  
Solución: SNMPv3.

### 4. Protocolos seguros vs inseguros

#### Inseguros:

- Telnet
    
- FTP
    
- HTTP
    
- SNMP v1/v2
    
- POP3
    
- IMAP sin TLS
    

#### Seguros:

- SSH
    
- SFTP / FTPS
    
- HTTPS
    
- SNMPv3
    
- LDAPS
    
- RDP con NLA
    

Pregunta típica:  
“¿Qué protocolo seguro reemplaza a Telnet?”  
→ SSH.

---

### 5. Modelo cliente-servidor

Concepto simple pero que Cisco evalúa:

- Cliente inicia la conexión.
    
- Servidor escucha en un puerto específico.
    
- Comunicación fluye según reglas del protocolo.
    

Ejemplo:  
Navegador → cliente  
Servidor web → servidor  
Puerto 443 → HTTPS

---

### 6. NAT (Network Address Translation)

Traduce direcciones privadas a públicas.  
No es un mecanismo de seguridad por sí mismo.  
Ayuda a:

- Ocultar IP internas
    
- Reducir uso de direcciones públicas
    

---

### 7. VPNs (Virtual Private Networks)

Cifran tráfico a través de Internet.  
Dos tipos básicos:

- Remote Access VPN
    
- Site-to-Site VPN
    

Proveen:

- Confidencialidad
    
- Integridad
    
- Autenticación
    

---

### 8. IPS/IDS relacionados con protocolos

- IDS detecta tráfico sospechoso (ej: DNS tunneling).
    
- IPS bloquea ataques basados en firmas/protocolos.

## 4. Resumen estratégico para memorizar

- TCP = confiable; UDP = rápido; ICMP = diagnóstico.
    
- HTTP = inseguro, HTTPS = seguro.
    
- DNS = objetivo común de ataques.
    
- DHCP = riesgo de servidores rogue.
    
- FTP/Telnet = no usados por ser inseguros.
    
- SSH/SFTP = acceso remoto + transferencia segura.
    
- SNMPv3 = seguro; v1/v2 = inseguro.
    
- NTP = sincronización; importante para logs.
    
- RDP = acceso remoto Windows.
    
- VPN = cifrado de extremo a extremo.
    
- NAT = traduce IP, no reemplaza seguridad.
